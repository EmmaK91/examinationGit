## 1) Egen user story + Acceptance Criteria + INVEST

**Story:**  
Som användare vill jag få en varning om jag försöker lägga till en tom uppgift, så att listan inte fylls med saker utan innehåll.

**Acceptance Criteria (checkboxar):**
- [x] Om inputfältet är tomt och jag trycker på “Lägg till”, ska ett felmeddelande visas och uppgiften ska inte läggas till.
- [x] När jag börjar skriva i inputfältet igen, ska felmeddelandet försvinna.
- [x] Felmeddelandet ska synas tydligt (t.ex. röd text).

**INVEST – kort motivering:**  
- **Independent:** Kan göras utan att röra andra funktioner.  
- **Negotiable:** Texten kan ändras senare utan problem.  
- **Valuable:** Gör appen mer användbar och tydlig.  
- **Estimable:** Går att göra på kort tid.  
- **Small:** Små ändringar i JS och eventhantering.  
- **Testable:** AC går enkelt att testa manuellt.

---

## 2) Sprintmål + Definition of Done (DoD)

**Sprintmål:**  
Få todo-listan att funka med validering, så att man kan lägga till, ta bort och markera uppgifter utan problem.

**DoD (checklista):**
- [x] Kod kör utan fel  
- [x] PR granskad och godkänd  
- [x] README uppdaterad  
- [x] Felmeddelande fungerar  
- [x] LocalStorage sparar uppgifter  

---

## 3) Retro: Start / Stop / Continue

- **Start:** Ha bättre koll på vilken branch jag är i när jag gör ändringar, så slipper jag göra om allt.  
- **Stop:** Övertänka allting och vara rädd för att ha "för många" brancher. Bättre att ha fler små PR än få stora.  
- **Continue:** Använda feature-brancher, PR med squash och merge.  

**Förbättring:** Nästa gång vill jag ha ett tydligare mål innan jag börjar med projektet. Se till att ha alla kort klara och tydliga, istället för att ändra, lägga till nya och ta bort för att det inte hinns med.
